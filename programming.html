gcc, cc, g++, c++

are used to invoke a GNU C and C++ compilers.

g++ (c++) is a program that calls gcc (cc) with the default language set to C++, and automatically specifies linking against the C++ library. In most cases options are the same.

cc app.c

build an executable file from the source file app.c;

cc -Wall app.c -o apptest

build an executable file apptest from the source file app.c, use the typical options to generate warning messages;

c++ -O2 -Wall app.cpp main.cpp -o conv4

build conv4 from app.cpp and main.cpp, use reasonable options for code optimization and warning msgs;

c++ chkpass.cpp -o chkpass -lldap

build chkpass program with LDAP support;

cc -v

show miscellaneous info about gcc (version, features, etc);

cc -v -c app.c

create an object file verbosely (i.e., output detailed info about cmds executed, their versions, etc);

cc -E app.c > app.c.prep

create a preprocessed file (may be large);

cc -E app.c -dM | sort | less

show all #define macros in app.c;

cc -S app.c

convert the C code into assembly language;

cc -M app.c

generate Makefile dependencies;

cc -g -o helloworld app.c

build binary executable file with additional debugging information (all optimization should be turned off);

cc -pg -o helloworld app.c

build a binary executable file with additional profiling info;

Compilation can involve up to 4 stages:

  • preprocessing,
  • compilation,
  • assembly,
  • linking

(always in that order)

~ is capable of preprocessing and compiling several files either into several assembler input files, or into one assembler input file; then each assembler input file produces an object file, and linking combines all the object files into an executable file. ~ normally does preprocessing, compilation, assembly and linking, but there are options to stop this process at an intermediate stage. ~ accepts options and file names as operands. Many options have multi-letter names; therefore multiple single-letter options may not be grouped: -dr is very different from -d -r. You can mix options and other arguments. For the most part, the order doesn’t matter. However, it does matter when you use several options of the same kind (-I or -L specified more than once: the dirs are searched in the order specified).

Many options have long names starting with -f or -W. Most of these have both positive and negative forms (-ffor-scope and -fno-for-scope).

Some options are overall (-c, -o, -v, -x), some options are related to preprocessor (-I, -D), some to linker (-L, -l).

Search path for #include

If stmt looks like

#include "file"

the curr dir is searched first, then dirs specified by -iquote options, then the same places as for angle brackets.

If stmt looks like:

#include <file>

the search order is:

/usr/local/include
libdir/gcc/target/version/include/c++/version
libdir/gcc/target/version/include
/usr/target/include
/usr/include

The second line is for C++ programs only. The target is the canonical name of the system GCC was configured to compile code for; often but not always the same as the canonical name of the system it runs on.

Note!

In this context canonical name is something like 'cpu-company-system', e.g., x86_64-linux-gnu or i686-linux-gnu. Wikipedia defines canonicalization (aka "standardization" or "normalization") as "a process for converting data that has more than one possible representation into a standard, normal, or canonical form".

All dirs added by -I are searched in left-to-right order before the default dirs. However, if the specified dir is already searched by default, the option is ignored and the search order for system dirs remains unchanged. To prevent searching any of the default dirs, use -nostdinc option.

Notes on C/C++ libraries

There are two C/C++ library types in Linux:

  1. Static libs (*.a): the libraries of object code that are linked with the application, and become the part of the application [sometimes essentially increasing the size of the executable file].
  2. Dynamically linked shared object libs (*.so). There is only one form of these libraries but it can be used in two ways:
    • Dynamically linked at run time but statically aware. The libraries must be available during compile/link phase. The shared objects are not included into the executable component but are tied to the execution.
    • Dynamically loaded/unloaded and linked during execution (e.g., browser plug-in) using the dynamic linking loader system functions.
How to generate a static library

Let's assume that I have 2 (3, 4, 5,..) functions in separate files and I want to create a library. The steps are:

cc -Wall -c cfunc1.c cfunc2.c

ar -cvq libctest.a cfunc1.o cfunc2.o

To list files in this library:

ar -t libctest.a

To compile an app (test1) with this lib:

cc -o test1 test1.c -lctest

Also, -Lpath_to_lib_dir may be required.

The Linux .a lib is similar to Visual C++ static .lib.

How to create a shared library (dynamically linked object library)
  • Create object code;
  • Create library;
  • Optional step: create default version using a symbolic link;

For example:

gcc -Wall -fPIC -c *.c

gcc -shared -Wl,-soname,libctest.so.1 -o libctest.so.1.0 *.o

mv libctest.so.1.0 /usr/local/lib

ln -sf /usr/local/lib/libctest.so.1.0 /usr/local/lib/libctest.so.1

ln -sf /usr/local/lib/libctest.so.1.0 /usr/local/lib/libctest.so

Compiler options:

  • -Wall: include warnings;
  • -fPIC: compiler directive to output position independent code (required by shared libs; also -fpic);
  • -shared: create a shared obj which can then be linked with other objects to form an executable;
  • -W1: pass options to linker (the options to be passed on to the linker are: -soname libctest.so.1);
  • -o: output (in this case the name of the shared object libctest.so.1.0);
Versioning

Library versions should be specified for shared objects if the function interfaces are expected to be changed (functions added/removed, func prototypes changed, obj definitions changed, etc).

If the library is expected to be updated, then a lib version should be specified when the shared obj lib is created. This is especially important for shared obj libs that are dynamically linked. This also avoids the "so called" Microsoft DLL hell, when a system upgrade changes std libs and thus breaks the older apps requiring the older versions of the shared obj functions.

Versioning occurs with the GNU C/C++ libs too, and this often makes binaries compiled with one version of the GNU tools incompatible with binaries compiled with other versions unless those versions also reside on the system. Versioning allows multiple versions of the same lib to co-exist on the same system peacefully. The version of the lib is included in the symbol name, so the linker knows which version to link with.

Symbol types in object files
A The symbol's value is absolute, and will not be changed by further linking.
B
b		 The symbol is in the uninitialized data section (known as BSS).
C The symbol is common (uninitialized data). When linking, multiple common symbols may appear with the same name. If the symbol is defined anywhere, the common symbols are treated as undefined references.
D
d		 The symbol is in the initialized data section.
G
g		 The symbol is in an initialized data section for small objects. Some obj file formats permit more efficient access to small data objects, such as a global int var as opposed to a large global array.
i For PE format files it indicates that the symbol is in a section specific to the implementation of DLLs. For ELF it indicates that the sym is an indirect function. This is a GNU extension to the std set of ELF symbol types. It indicates a symbol which if referenced by a relocation does not evaluate to its addr, but instead must be invoked at runtime. The runtime execution will then return the value to be used in relocation.
N Debugging symbol.
p The symbol is in a stack unwind section.
R
r		 The symbol is in a read only data section.
S
s		 The symbol is in an uninitialized data section for small objects.
T The symbol is in the text (code) section.
U The symbol is undefined.
u The symbol is a unique global sym. This is a GNU extension to the std set of ELF sym bindings. For such sym the dynamic linker will make sure that in the entire process there is just one sym with this name and type in use.
V
v		 The symbol is a weak object. When a weak defined sym is linked with a normal defined sym, the normal defined sym is used with no error. When a weak undefined sym is linked and the sym is not defined, the value of the weak sym becomes zero with no error. On some systems, uppercase indicates that a default value has been specified.
W
w		 The symbol is a weak sym that has not been specifically tagged as a weak obj sym. When a weak defined sym is linked with a normal defined sym, the normal defined sym is used with no error. When a weak undefined symbol is linked and the sym is not defined, the value of the sym is determined in a system-specific manner without error. On some systems, uppercase indicates that a default value has been specified.
- The symbol is a stabs sym in an a.out obj file. In this case, the next values printed are the stabs other field, the stabs desc field, and the stab type. Stabs symbols are used to hold debugging info.
? The symbol type is unknown, or object file format specific.

A weak symbol is a symbol definition in an obj file or dynamic lib that may be overridden by other sym definitions.

JAR

JAR (Java ARchive) is an archive file format typically used to aggregate many Java class files and associated metadata and resources into a single file for convenient distribution. JAR is actually ZIP archive, but you should use Java tools to work with it.

To run a Java application:

java -jar TableDemo.jar

To create a JAR including class files, dirs audio and images:

jar cvf SoundDemo.jar *.class audio images

Also, you can put in archive all appropriate files and dirs, including source, comments, READMEs, etc (note that META-INF dir and manifest are created automatically; you can modify manifest afterwards):

jar cvf SoundDemo.jar *

To view the contents of a JAR file:

jar tvf SoundDemo.jar

To extract contents of a JAR file:

jar xvf SoundDemo.jar

In both cases v (verbose) is optional.

To update an existing JAR file (i.e., to add/replace files, dirs):

jar uf SoundDemo.jar images/app_logo.gif

When you create a JAR file, a default manifest is created automatically as META-INF/MANIFEST.MF. There can be only one manifest in archive!

An example of a simple manifest file:

Manifest-Version: 1.0

Created-By: 1.6.0_02-b06 (Sun Microsystems Inc.)

Main-Class: SoundDemo

Product-Name: Sound Processing App
To modify manifest, use m option and a separate text file with lines to be added (format is key:value, see examples below and an important requirement):

jar ufm SoundDemo.jar manif.txt

To modify the default manifest while creating a new JAR file:

jar cfm fname.jar manifest-add in-files

Options and arguments:

  • c indicates that you want to create a JAR file;
  • f indicates that you want the output to go to a fname.jar (rather than to stdout);
  • m indicates that you want to merge info from the specified file into the manifest of the archive you're creating;
  • manifest-add is the name (path/name) of a text file whose contents you want to add to the manifest; note that:
    1. This file must have NL or CR at the end!
    2. The contents must be encoded in UTF8.
  • in-files is a space-separated list (or pattern) of files that must be placed in the JAR file.

Note that m and f must go in the same order as the corresponding files.

To add an application's entry point (i.e., class having main() function) to manifest, put the following line in manif.txt:

Main-Class: TableDemo

After that:

jar cfm TableDemo.jar manif.txt *.class

If you need to reference classes in other JAR files:

Class-Path: MyUtils2.jar libs/AbcLib4.jar

Zenity

With zenity you can provide some basic GUI features in shell scripts. It displays miscellaneous GTK+ dialogs, and returns user's input (either in the return code or on stdout). For example:

#!/bin/bash

if zenity --info --text="The time is
    `date +%R`; Pause and relax!"; then

  exit 0
fi

This script can be run by cron and would remind you each hour to have a rest. The appropriate crontab line can look like this:

50 8-17 * * * env DISPLAY=:0 /home/pro/bin/time_warn.sh

See crontab for explanations.
